# Report Extra Details: Training, Evaluation, and Demo Evidence

This file is a report-support note for the final ENPM690 writeup. It collects implementation details that are easy to lose in the project narrative: mixed-start training, reward definitions, success thresholds, Gazebo demo evidence, and concrete repository changes.

The intended project claim remains narrow and honest:

> This project demonstrates a modular L1-to-RL manipulation prototype with a working kinematic `Approach -> Finisher` skill stack, a Qwen/L1 semantic bridge, Gazebo/RViz demo runtime support, and route/workspace curriculum extensions. It does **not** claim full holder1-to-holder8 kitchen transport is solved.

### Start-State Pool

The start-state pool is generated by `hrl_ws/src/hrl_trainer/hrl_trainer/kinematic_phase1/workspace/workspace_start_state_map.py`.

Each start sample records:

| Field | Meaning |
|---|---|
| `start_id` | Unique start-state identifier |
| `q_start` | Initial 7-DOF joint state |
| `dq_start` | Initial joint velocity proxy |
| `prev_action` | Previous normalized action vector |
| `ee_position` | FK end-effector position |
| `ee_orientation` | FK end-effector orientation representation |
| `source_type` | Source category |
| `source_stage` | Curriculum stage if applicable |
| `source_rollout_id` | Synthetic or rollout provenance ID |
| `stability_score` | Joint-limit and motion stability proxy |
| `joint_limit_margin_min` | Minimum normalized joint-limit margin |
| `bucket_id` | Workspace bucket ID |

Start-state sources:

| Source | Purpose |
|---|---|
| `home` | Preserve the original home-start behavior |
| `successful_rollout` | Replay known good states from Stage 0-7 and stable expansion rollouts |
| `near_target` | Train recovery and settling near target states, especially in harder stages |
| `random_valid_q` | Break the dependency on always starting from home |
| `failure_recovery` | Revisit failed or difficult states with small perturbations |
| `stress_state` | Limited exposure to difficult reachable configurations |

| Number of stages used | `12` (`0-11`) |
| `start_random_samples` | `384` |
| Nominal generated start samples | `1 + 64*12 + 384 = 1153` |

### Start-Target Pair Difficulty

The start-target pair sampler is implemented in `hrl_ws/src/hrl_trainer/hrl_trainer/kinematic_phase1/workspace/start_target_pair_sampler.py`.

Each pair computes:

| Quantity | Purpose |
|---|---|
| `joint_distance_l2` | Main distance in joint space |
| `ee_position_distance` | Cartesian end-effector distance |
| `orientation_distance` | Pose orientation gap |
| `z_displacement` | Vertical transfer difficulty |
| `joint_limit_margin_min` | Safety / difficulty near joint limits |
| `target_previous_success_rate` | Adaptive frontier signal |

Default pair classes:

| Class | Definition |
|---|---|
| `retention` | Known good starts / targets, mostly Stage 0-7 |
| `local` | `joint_distance_l2 <= 0.28` |
| `medium` | `joint_distance_l2 <= 0.70` |
| `frontier` | Target bucket has prior success between `0.35` and `0.80`, or target is Stage <= 10 but beyond medium |
| `stress` | Very low prior success or harder Stage 9-11 / random reachable pairs |

Configured pair count:

| Setting | Value |
|---|---:|
| `pair_count` | `2048` |
| Minimum pair joint L2 | `0.04` |

### Bucket Definition and `covered_bucket_fraction = 0.200`

Workspace buckets are not physical shelves. They are sampled-region bins used to estimate coverage of the reachable set.

The target bucket ID is constructed from:

```text
x bucket, y bucket, z bucket, orientation bucket, joint-L2 bucket
```

Implementation summary:

| Bucket dimension | Definition |
|---|---|
| `x/y/z` | FK position components bucketized into 8 bins over the sampled target map span |
| `orientation` | Norm of orientation vector bucketized into 6 bins over `[0, pi]` |
| `joint_l2` | `||q_target||` bucketized into 6 bins over `[0, 4.5]` |
| `bucket_id` format | `x{xb}_y{yb}_z{zb}_o{ob}_q{qb}` |

For the final random-start coverage eval:

| Coverage metric | Value |
|---|---:|
| Observed eval buckets | `35` |
| Covered bucket count | `7` |
| Covered bucket fraction | `0.200` |
| Stable bucket fraction | `0.0286` |
| Partial bucket fraction | `0.1714` |
| Stress bucket fraction | `0.800` |

Interpretation: `covered_bucket_fraction = 0.200` means that, among buckets actually exercised during the final random-start coverage evaluation, 20% had successful samples. It is **not** a claim that 20% of the robot's continuous physical workspace is solved.

| Finding | Meaning |

| Position dominates failure in hard splits | The large-workspace bottleneck is mostly distance / reachability / accumulation, not just final orientation. |

## 2. Reward Configuration Details

The final policy stack uses two main reward profiles:

| Policy | Config |
|---|---|
| Approach / workspace expansion | `hrl_ws/src/hrl_trainer/hrl_trainer/kinematic_phase1/configs/workspace_expansion_dynamic_scale_big.yaml` |
| Finisher / hold-and-preserve | `hrl_ws/src/hrl_trainer/hrl_trainer/kinematic_phase1/configs/dock_workspace_handoff_noop_ft_12env.yaml` |

### Approach Reward Summary

| Reward part | Purpose | Main terms / actual values |
|---|---|---|
| Position progress | Move toward the target | `position_progress_weight = 7.0` |
| Far-field action scaling | Permit larger moves far from target | `action_delta_scale = 0.08`, far multiplier `1.75` when position error >= `0.060 m` |
| Orientation alignment | Coordinate pose once close enough | global `orientation_progress_weight = 0.3`, near-field `3.8` |
| Near-goal zone | Detect precise handoff region | `near_goal_pos_threshold_m = 0.005`, `near_goal_ori_threshold_rad = 0.05` |
| Finisher readiness | Train same-step dockable state | pos `<= 0.005 m`, ori `<= 0.05 rad`, action `<= 0.1`, dq `<= 0.002` |
| Readiness bonus | Encourage entering handoff state | `finisher_ready_bonus = 3.2` |
| Readiness retention | Encourage staying handoff-ready | `finisher_ready_retention_bonus = 2.7`, `finisher_ready_dwell_bonus = 3.7` |
| Leave / regression penalty | Prevent losing a good state | `finisher_ready_leave_penalty = 4.2`, `finisher_ready_regression_weight = 18.0` |
| Near-handoff low motion | Quiet handoff | action weight `2.0`, dq weight `8.0`, motion bonus `4.0`, settle bonus `4.0` |
| Same-step alignment | Synchronize position and orientation | `same_step_alignment_bonus = 0.12` |
| Drift / smoothness | Avoid unstable control | `drift_penalty_weight = 2.5`, action magnitude `0.04`, action delta `0.045` |
| Joint safety | Avoid joint-limit exploitation | `joint_limit_penalty_weight = 0.06` |

Zone-aware design:

| Zone | Dominant reward behavior |
|---|---|
| Far field | Position progress and larger action scale dominate. Orientation is intentionally light. |
| Mid field | Position and orientation progress both matter. Regression penalties increase. |
| Near handoff | Same-step readiness, dwell, low motion, and leave/regression penalties dominate. |

### Finisher Reward Summary

| Reward part | Purpose | Main terms / actual values |
|---|---|---|
| Tiny correction | Allow small final adjustment | position progress `0.12`, orientation progress `0.12` |
| Strict zone | Define final hold region | pos `<= 0.005 m`, ori `<= 0.05 rad` |
| Strict hold | Stay successful once there | `strict_hold_bonus = 0.55`, strict center dwell weight `0.50` |
| Preserve state | Do not ruin a good Approach handoff | `preserve_state_bonus = 0.80` |
| Worse-than-entry penalty | Penalize degrading the handoff state | position weight `85.0`, orientation weight `8.0` |
| Low motion | Quiet final control | `low_motion_bonus = 0.90`, action threshold `0.035`, dq threshold `0.0015` |
| Aggressive action penalty | Prevent the frozen finisher failure mode | `aggressive_action_weight = 1.10`, threshold `0.055` |
| Near-strict penalties | Stronger penalties near success | regression multiplier `3.0`, action/dq multipliers `3.8` |
| Smoothness | Avoid jitter | action magnitude `1.25`, action delta `1.60` |
| Joint safety | Avoid invalid postures | `joint_limit_penalty_weight = 0.05` |

The finisher is intentionally a hold-and-preserve policy, not a large-range approach policy.

## 3. Success Definitions

Different experiments use different success definitions. This is important because `success_rate` does not always mean the same thing.

| Context | Definition | Notes |
|---|---|---|
| Approach / workspace stage success | `pos <= 0.005 m`, `ori <= 0.05 rad`, dwell `>= 2` | Main kinematic skill-stage eval. |
| Finisher-ready reward condition | `pos <= 0.005 m`, `ori <= 0.05 rad`, `action <= 0.1`, `dq <= 0.002` | Used to train same-step clean handoff behavior. |
| Strict finisher success | `pos <= 0.005 m`, `ori <= 0.05 rad`, dwell `>= 5` | Used by the hold-and-preserve finisher. |
| Demo handoff trigger | `pos <= 0.005 m`, `ori <= 0.20 rad`, confirm `1` | Gazebo demo threshold; relaxed in orientation so the screen recording does not wait on invisible millimeter/radian details. |
| Demo relaxed strict success | `pos <= 0.015 m`, `ori <= 0.20 rad` | Presentation/demo status only, not the strict research metric. |
| Route waypoint success | `pos <= 0.010 m`, `ori <= 0.150 rad`, `q_error <= 0.500`, action `<= 1.20`, dq `<= 0.040`, dwell `>= 1` | Route-prefix curriculum metric. |
| Qwen/L1 demo success | Valid object/source/target/constraint resolution and skill request generation | Semantic contract validation, not physical manipulation success. |
| Gazebo live demo success | L1 resolves intent, marker is shown, runtime publishes L2/L3 statuses, arm moves under controlled policy runtime | Execution-path evidence, not full tray transport completion. |

The report should state which success definition is used for each table or plot.

## 4. Gazebo / RViz Demo Evidence

The live demo is meant to show the integration path, not to claim full physical tray transport.

Demo chain:

```text
Natural language command
-> Qwen / fallback L1 bridge
-> IntentPacket
-> APPROACH -> FINISHER skill request
-> RViz/Gazebo target markers
-> controlled RL runtime / L2-L3 status
-> arm motion in Gazebo
```

Canonical demo command:

```text
Move tray1 from shelf_A1 to shelf_B1 while keeping it level and inserting with a stable pose.
```

Resolved L1 fields:

| Field | Value |
|---|---|
| `object_id` | `tray1` |
| `source_slot` | `shelf_A1` |
| `target_slot` | `shelf_B1` |
| `speed_cap` | `SLOW` |
| Pipeline | `APPROACH -> FINISHER` |
| Target xyz | `[-0.92, -1.16, 1.22]` |
| Target rpy | `[3.14, 0.0, 3.14]` |

Tray-like semantic waypoint plan used for demonstration:

```text
pre_grasp_align
-> under_tray_insert_pose
-> level_lift
-> carry_midline
-> pre_insert_align
-> stable_insert_hold
```
